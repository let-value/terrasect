# Multiversion support

Terrasect builds one shared codebase against several Minecraft versions and both
loaders. Two independent axes:

- **Loader** (Fabric / NeoForge) — handled by separate modules (`fabric/`,
  `neoforge/`) plus loader constants. Not the concern of this document.
- **Minecraft version** — handled by [Stonecutter](https://stonecutter.kikugie.dev/)
  source preprocessing plus a small compat shim layer. This document is about the
  version axis.

## Target matrix

| MC version | Java | API world | Notes |
|---|---|---|---|
| `1.21.1`   | 21 | old | back-compat target |
| `1.21.11`  | 21 | new | primary / active dev version |
| `26.1`     | 25 | new | MC `26.1.2` |
| `26.2`     | 25 | new | latest |

Each version is built for both `fabric` and `neoforge`, so the full matrix is
4 × 2 = 8 buildable variants (plus a `-common` project per version).

Per-version dependency coordinates live in `stonecutter.properties.toml`; the
matrix itself is declared in `settings.gradle.kts`. The active dev version is set
by `stonecutter active "..."` in `stonecutter.gradle.kts`.

Gradle exposes each variant as its own project: `:<version>-<loader>`, e.g.
`:1.21.11-fabric`, `:26.1.x-common`. The `versions/` directory holding these is
generated by Stonecutter and git-ignored.

## The single fault line: `>=1.21.11`

Every version-divergent API in the codebase splits on **one** boundary:
`//? if >=1.21.11`. There are only two API worlds:

| concern | old (`1.21.1`) | new (`>=1.21.11`, incl. 26.x) |
|---|---|---|
| resource key id | `key.location()` | `key.identifier()` |
| spawner entry type | `SpawnerData.type` | `SpawnerData.type()` |
| loot param lookup | `getParamOrNull(...)` | `getOptionalParameter(...)` |
| surface density fn | `initialDensityWithoutJaggedness` | `preliminarySurfaceLevel` |
| structure → key | `listElements().filter(...)` | `getResourceKey(...)` |
| debug overlay | *(absent)* | `DebugScreenEntry` API |

`26.1` and `26.2` share the `1.21.11` (new) branch — they need no separate
predicate. If a future version diverges again, add a second boundary rather than
widening this one.

## Where divergence lives

Keep version `//?` predicates out of business logic. Route each divergent call
through a named shim in `common/src/main/kotlin/terrasect/compat/`:

- `ResourceKeyCompat` — key id / path (by far the most-used shim)
- `SpawnCompat` — spawner entry type
- `LootContextCompat` — loot context origin
- `NoiseRouterCompat` — surface density function + its key

Business code calls e.g. `ResourceKeyCompat.getKeyId(key)` and stays
version-clean. Only three sites legitimately keep an inline predicate because the
whole construct differs, not just one call: `mixin/structure/StructureStartMixin.java`,
`gui/RegionDebugEntry.kt` (the entire class depends on the new `DebugScreenEntry`
API), and `TerrasectFabricClient.kt` (registers that entry).

Shim shape (note the `spotless:off/on` fences — Stonecutter comments must survive
formatting, and the inactive branch is a quoted comment):

```kotlin
// spotless:off
//? if >=1.21.11 {
return key.identifier().toString()
//?} else {
/*return key.location().toString()
 */
//?}
// spotless:on
```

## Access wideners

One per version world under `common/src/main/resources/accesswideners/`, selected
at build time by version string. `1.21.11` uses the `named` namespace; the modern
`26.1` / `26.2` wideners use `official` (newer loom intermediary) with identical
entries.

## Adding a version

1. Add a `match(...)` line in `settings.gradle.kts`.
2. Add `[...]`, `[fabric...]`, `[neoforge...]` sections in
   `stonecutter.properties.toml` with real coordinates (NeoForge/NeoForm from
   `maven.neoforged.net`, Fabric API/loader from `maven.fabricmc.net`).
3. Add `<version>.accesswidener` (copy the nearest same-era one).
4. If it diverges from the existing API worlds, add a compat shim / predicate;
   otherwise it inherits the matching branch for free.
5. Verify: `./gradlew :<version>-fabric:build :<version>-neoforge:build`.
