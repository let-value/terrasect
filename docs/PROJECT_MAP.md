# Terrasect Project Map Document

Terrasect's core goal is to enable predictive world generation by establishing a **narrative world partitioning** system. This system registers relationships (`RegionRegistry`) between macro-level regions, allowing the generation of consistent, traversable, and structured worlds, while strictly adhering to performance and code clarity guidelines.

## 📂 Directory Structure

The project is designed with a clear separation of concerns mandated by the respective Minecraft mod loaders, keeping core logic centralized and minimizing platform-specific code in hot paths.

### `/home/alex/terrasect/common/`: Core Logic (Platform Agnostic)
This is the primary repository for business logic, data structures, and generation mechanics. It defines *what* needs to be generated without caring *how* (client/server).
*   **`com.terrasect.common.definition`**: Defines the fundamental data model:
    *   `RegionDefinition`: Holds the complete rule set for a region (Climate, Height, Noise, Biomes, Structures, Mobs).
    *   `Region`: Represents an instantiated region, detailing its physical properties (area, radius, neighbors) and relationships.
*   **`com.terrasect.common.generation`**: Contains the world construction logic.
    *   `World`: The main entry point for world initialization, handling the registration and scaling of regions based on dimension ID and seed.
    *   `Layout`: Handles the traversal path finding, determining which region contains a given coordinate $(x, z)$.
*   **`com.terrasect.common.handler`**: Contains dedicated logic for applying constraints (e.g., `ClimateHandler`, `NoiseHandler`, `BiomeHandler`).
*   **`com.terrasect.common.util`**: Utility classes (`MathUtils`, `NoiseUtils`, `Packer`) for reusable mathematical and data handling operations.
*   Mixin Classes: Provide accessors (`*Accessor.java`) to intercept vanilla Minecraft methods for modification.

### `/home/alex/terrasect/fabric/`: Fabric-Specific Code
Contains all Mixins and setup code specific to the Fabric loader environment. These Mixins intercept Minecraft's vanilla methods (e.g., `Climate.Sampler.sample`) to insert custom generation logic.

### `/home/alex/terrasect/neoforge/`: NeoForge-Specific Code
Contains all equivalent Mixins and setup code specific to the NeoForge loader environment. They mirror the functionality of the Fabric Mixins but conform to the NeoForge API.

## 🔑 Key Classes

| Class | Location | Purpose |
| :--- | :--- | :--- |
| **`Region`** | `common/definition` | The physical representation of a generated section of the world. It holds an ID, geometric parameters (radius, budget), and a reference to its rules (`RegionDefinition`). |
| **`RegionDefinition`** | `common/definition` | A value object that encapsulates all the rules governing a region's generation (e.g., which biome generation rules to use, how height must change, etc.). |
| **`World`** | `common/generation` | Acts as the global world state manager. It initializes the dimension-specific `DimensionState` by finding the correct region hierarchy and calculating coordinate offsets. |
| **`ClimateMixin`** | `common/mixin` | Intercepts the vanilla `Climate.Sampler` to execute `ClimateHandler` logic, modifying the generated target point based on Terrasect's rules. |
| **`VanillaSamplerAccessor`** | `common/mixin` | Provides a controlled, synchronized means for Mixins to sample the vanilla climate system when implementing Terrasect modifications. |
| **`DimensionState`** | (Internal to `World`) | Tracks the state of a specific dimension (e.g., Overworld) within the world save, storing the root region, noise registry, and calculated grid offsets. |
| **`ClimateHandler`** | `common/handler` | Centralized utility responsible for modifying the target point generated by the Minecraft climate system. |

## 🔁 Data Flow (Region Lifecycle)

The generation process follows a strict, chained workflow:

1.  **Registration:** The root region (e.g., the global Overworld region) is registered via `World.register()`. This builds the initial `CompiledNoiseRegistry`.
2.  **Initialization (`World.initialize`):** When the game attempts to stream or generate chunks for a specific dimension/seed, `World.initialize()` is called.
    *   It retrieves the `DimensionState` based on the dimension ID and seed.
    *   It calculates if the world needs re-initialization.
    *   It identifies and calculates the necessary grid offsets ($\Delta x, \Delta z$) required to correctly map the region's anchor point relative to the world origin.
    *   The `DimensionState` is updated with the offsets.
3.  **Traversal and Query:** When coordinates $(x, z)$ are needed:
    *   A `TraversalIterator` (within the `Layout` class) is used to find the correct `Region` corresponding to those coordinates, factoring in the dimension's offsets.
    *   The iterator steps through the region hierarchy until the container region is found.
4.  **Generation Execution:**
    *   Once a `Region` and its associated `RegionDefinition` are known, the generation pipeline executes the rules defined in the `RegionDefinition`:
        *   **Climate/Noise (Hot Path):** Mixins like `ClimateMixin` intercept vanilla calls. The associated `Handler` (e.g., `ClimateHandler`) uses the `MinecraftContext` to read the current state and modify the output ($\text{TargetPoint}$).
        *   **Structures/Biomes:** Specialized Handlers use `SelectionRules` and `Lookups` to validate and place discrete world elements within the constraints of the region.

## ⚠️ Core Assumptions & Constraints

1.  **Performance Priority (Hot Path):** The system assumes that worldgen/render-adjacent code (per-block or per-chunk) must be *allocation-free* and extremely minimal. Use of `ThreadLocal` or static methods is preferred to avoid per-chunk garbage collection overhead.
2.  **Modularity (Separation of Concerns):** The code strictly maintains logical boundaries: `common` for rules, and `fabric`/`neoforge` for system hooks. Platform-specific code should not contain domain logic.
3.  **Explicit State:** State management (especially offsets and initialization) is highly explicit, relying on `volatile` fields within `DimensionState` and checking initialization flags to prevent race conditions or stale data.
4.  **Predictability:** The entire architecture is built around the concept of *predictable journeys*. The region registry and traversal logic guarantee that any coordinate query resolves predictably to a region, regardless of how many times the world is loaded or which direction the traversal path progresses.
5.  **Readability Over Abstraction:** The style guide dictates avoiding "frameworks inside a mod" and prefers concrete, easily traceable data flow, even if it results in some boilerplate (e.g., custom Builder patterns for `RegionDefinition`).
