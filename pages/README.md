# pages/

The GitHub Pages site for Terrasect, and the single source of truth for user-facing project copy —
this is what mod users see, not contributors. Nothing in here should describe implementation
details (mixins, packages, internal APIs); that belongs in `docs/` instead.

Deployed automatically to GitHub Pages when a release tag is pushed (see
`.github/workflows/pages.yml`).

## Layout

```
pages/
├── build.sh                # Renders content/*.md with pandoc into dist/index.html (one page, anchors)
├── style.css                # Modern CSS: native light/dark theming via light-dark(), no manual toggle
├── images/
│   ├── regions/              # Region-boundary PNGs from terrasect.strategies.*Test
│   └── decorations/           # Decoration effect PNGs from DecorationGalleryTest
├── content/
│   ├── summary.txt          # One-line plain-text summary — mirrors Modrinth's "Summary" field
│   ├── description.md       # Long-form overview — mirrors Modrinth's "Description"/body field
│   ├── getting-started.md   ┐
│   ├── regions.md            
│   ├── decorations.md        
│   ├── noise.md               One section per page, in this order, stitched by build.sh
│   ├── archetypes.md         
│   ├── structures.md        
│   ├── mobs.md               
│   ├── loot.md               
│   └── commands.md          ┘
├── dist/                   # Generated by build.sh — git-ignored, not committed
└── README.md                # This file
```

Only `summary.txt` and `description.md` are shared with Modrinth (see below) — the rest of
`content/` only ends up in the stitched GitHub Pages site, since Modrinth's project page doesn't
support additional linked pages.

## One page, built with pandoc

Everything under `content/` gets rendered by [pandoc](https://pandoc.org/) — a widely available
CLI Markdown renderer (preinstalled on GitHub's `ubuntu-latest` runners; `brew install pandoc` /
`apt install pandoc` locally) — and stitched into a single `dist/index.html`, one `<section id="…">`
per file, instead of separate pages. An earlier version of this site shipped its own hand-rolled
JS Markdown renderer; it had rendering bugs pandoc doesn't have, so it's gone.

```bash
./pages/build.sh          # writes pages/dist/index.html (+ copies style.css and images/)
open pages/dist/index.html
```

Cross-references between sections use anchors (`[Regions](#regions)`, or `[x](#some-heading-id)`
for a specific heading — pandoc slugs headings the same way GitHub does, e.g. `### Forcing a
structure` → `#forcing-a-structure`). `pages/build.sh`'s `SECTIONS` array is the one place that
lists section order, ids, and nav labels — add a new `content/*.md` file there to add a section.

`content/summary.txt` and `content/description.md` are still structured the same way Modrinth
splits a project page (title, summary, description) so the same two files can be:

1. rendered into the `#overview` section of the stitched page, and
2. pushed as-is to Modrinth's project description via their API (see below),

without maintaining two copies of the same marketing copy.

## Regenerating the images

`images/regions/*.png` and `images/decorations/*.png` are copied straight from the mod's own
snapshot tests, not hand-drawn — see `common/src/test/kotlin/terrasect/strategies/*Test.kt`
(`DecorationGalleryTest` for the decoration effects). To refresh them after a strategy or decoration
changes:

```bash
./gradlew :26.2.x-common:test --tests "terrasect.strategies.*"
cp common/build/test-snapshots/26.2/strategies/HexStrategyTest/cells.png pages/images/regions/hex.png
cp common/build/test-snapshots/26.2/strategies/DecorationGalleryTest/warp.png pages/images/decorations/warp.png
# ...repeat per image, see git history of this file for the full copy list
```

## Publishing to Modrinth

`.github/workflows/pages.yml` has a `sync-modrinth` job that PATCHes `content/summary.txt` and
`content/description.md` straight to the Modrinth project (`PATCH /v2/project/{id}`) on every
release tag. It's `continue-on-error: true` and no-ops if the secrets aren't set. `MODRINTH_TOKEN`
needs the **Write projects** scope for this job (on top of whatever `publish.yml` already needs —
**Create versions**) — a token without it will get an auth error here.

## CurseForge

CurseForge does not expose a public API for editing a project's page/description — only for
uploading version files (which `publish.yml` already does). Updating the CurseForge listing means
manually pasting the contents of `content/description.md` into their editor, reformatted for their
rich-text/BBCode input (it doesn't accept raw Markdown either). There's no way around that today.
